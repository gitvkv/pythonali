# Developer Guide & Project Documentation - Python Ali

Welcome to the development and maintenance documentation for **Python Ali** (`pythonali.com`). This document explains the design choices, repository structure, and build pipeline to help you manage the website, add future courses, and troubleshoot issues.

---

## 1. Project Directory Structure
The workspace is organized into two main parts:
1. **The Generator Engine (`doc_engine_rtd`)**: Contains the scripts, configurations, and generated static pages.
2. **The Content Source (`project_21_repo`)**: A sibling directory containing the raw Markdown files (`theory_for_student` and `code_for_student`).

### File Mapping
```text
doc_engine_rtd/
├── .github/workflows/ci.yml # Automated deployment script to GitHub Pages
├── docs/                    # Dynamically generated Markdown (DO NOT edit directly)
│   ├── Getting Started/     # Compiled chapters & index.md roadmap
│   ├── stylesheets/         # Custom CSS (extra.css)
│   └── index.md             # Landing page with CTA and About Us info
├── generate.py              # Main Python script that runs the build pipeline
├── mkdocs.yml               # Configuration file generated by generate.py
└── site/                    # Locally built HTML pages (output of mkdocs build)
```

## 2. The Generation Pipeline
The site is built dynamically using a custom Python script:

1. **Clean**: `generate.py` completely deletes and recreates the `docs/` directory to prevent stale files from lingering.
2. **Scrape**: It scans `../project_21_repo` for flat Markdown files matching the pattern `chXX_XX_XX-*.md`.
3. **Parse**:
   - Extracts the **Category Breadcrumb** from Line 1 (`# 📁 Category`).
   - Extracts the **Lesson Title** from Line 2 (`## 📁 Title`).
   - Extracts the remaining content body for both Theory and Code.
4. **Merge & Tabify**: It formats the lesson into a tabbed layout:
   ```markdown
   # Lesson Title
   <span class="category-tag">🏷️ Category Tag</span>

   === "📖 Theory"
       [Theory Content]

   === "💻 Code"
       [Code Content]
   ```
5. **Roadmaps**: Creates an `index.md` inside each module folder (like `Getting Started`) containing a dynamically generated chapter list with clickable links.
6. **Homepage**: Writes the root `docs/index.md` containing the Hero copy, CTA button, and Team About Us section.
7. **Write Configs**: Generates the custom `extra.css` and the `mkdocs.yml` configuration (which includes the full sidebar `nav` hierarchy).

## 3. Key Design Choices (The "Why")

### Why the `readthedocs` Theme?
Previously, we tried heavily customizing the *MkDocs Material* theme. However, the Material theme utilizes a complex sidebar container grid that broke when we tried to force a Sphinx/RTD style layout. 
By switching to the native `readthedocs` theme, we got the structural layout of standard technical documentation (like Ansible's or Sphinx's documentation) natively without writing complex layout overrides.

### Why Option C (3-Level Sidebar)?
With 4 levels of hierarchy (Course -> Module -> Chapter -> Lesson), fully displaying them on the sidebar led to "Indentation Hell" where text wrapped and looked extremely cramped.
* **Option C** keeps the sidebar flat at 3 levels: `Course Name` -> `Module Folder` -> `Lesson Page`.
* **Sidebar Link Truncation**: A custom CSS rule prevents text from wrapping to a second line, truncating long names with a clean ellipsis (`...`).
* **Roadmap Landing Pages**: When a user clicks on a Module folder (e.g. *Getting Started*), it opens a beautiful Roadmap index page in the reading panel serving as a visual syllabus.

### Category Tags below H1
Instead of showing the raw folder category as a giant H1 header at the top of the file, the generator strips it and formats it as a small, light-grey tag (`🏷️ Category`) right below the main lesson header.

### 100% Screen Width Reading Panel
The default ReadTheDocs theme caps the reading width at `800px` for text readability. Since this is an engineering and coding website, we overrode this in `extra.css` to allow the reading panel to stretch **100% of the screen width**, giving wide code blocks and console outputs maximum horizontal space without wrapping.

### Adaptive Tabs & CSS Styling
* **The Tab Styling**: The default `readthedocs` theme does not have built-in styling for the tabbed containers generated by `pymdownx.tabbed` (rendering them as plain radio buttons). We added custom CSS overrides in `extra.css` to hide the raw radio controls and style the toggle labels as clean, modern tabs.
* **Three Interactive Modes**: When a code file is available for a topic, it compiles three interactive tabs:
  * `📚 All-in-One` (Default Active Tab): Displays both the theory and the code section on the same page, separated by a clean horizontal divider. This ensures the student gets the full picture immediately and never misses the code section.
  * `📖 Theory Only`: Shows only the theory section, with a helpful callout at the bottom guiding the student to switch views when they are ready.
  * `💻 Code Only`: Displays only the code section, giving code blocks and syntax full screen focus.
* **Adaptive Tab Compilation (Safe for Future Courses)**: The generator script dynamically checks if a matching code file exists. If the code file does not exist (e.g., for non-coding/pure theory subjects), it automatically skips tabs and compiles the page as a standard, flat, tab-less article page.

## 4. Search Engine Optimization (SEO) & Google Indexing

To ensure the website is fully visible and ranked on Google:
* **Sitemap Generation**: The generator automatically inserts `'site_url': 'https://pythonali.com'` into `mkdocs.yml`. During `mkdocs build`, this triggers the automatic generation of a `sitemap.xml` listing all pages.
* **Google Search Console**: To get indexed quickly, register `https://pythonali.com` in Google Search Console, verify the domain, and submit the URL of your sitemap: `https://pythonali.com/sitemap.xml`.

## 5. Hosting & Deployment

The site is hosted on **GitHub Pages** and proxied through **Cloudflare** for DNS and SSL caching.

### How GitHub Pages Deployment Works
Because the repository only contains the `doc_engine_rtd` directory and does **not** contain the raw content files (`../project_21_repo`), GitHub cannot compile the site from scratch.
Instead:
1. You run `python generate.py` and `python -m mkdocs build` locally. This compiles your static site into the `site/` folder.
2. You commit and push your changes (including the `site/` folder) to your `main` branch.
3. A GitHub Action (`.github/workflows/ci.yml`) triggers automatically, copies the `site/` folder, and pushes it directly to a separate `gh-pages` branch.
4. GitHub Pages serves the site directly from the `gh-pages` branch.

### Cloudflare DNS Configuration
To ensure `https://pythonali.com` remains secure and online:
- **Root Domain CNAME**: `@` points to `your-github-username.github.io` (Proxied - Orange Cloud).
- **Subdomain CNAME**: `www` points to `your-github-username.github.io` (Proxied - Orange Cloud).
- **SSL/TLS Mode**: Must be set to **Full** or **Full (Strict)** inside Cloudflare to prevent circular redirection loops.

## 5. Troubleshooting & Maintenance

### How to Add a New Course (e.g. "Java for Infrastructure")
1. Open `generate.py` and locate the `Java Course Placeholder` block.
2. Replace the hardcoded `coming_soon.md` with your generator loop targeting your Java folders, following the same logic as the Python build blocks.
3. Add the folders to the `GROUPS` mapping at the top of the script so they categorize correctly.

### Local Development / Testing
To test changes locally before pushing them live, run these commands inside your project folder:
```bash
# 1. Regenerate docs and mkdocs.yml
python generate.py

# 2. Start the local server on your custom port (8088)
python -m mkdocs serve -a localhost:8088
```
Go to `http://localhost:8088` in your browser. Any edits you make to your local Markdown files or generator script will refresh automatically.
